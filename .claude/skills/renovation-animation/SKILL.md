# Renovation Visualiser Teardown Animation

## What This Is

An enhanced before/after animation for the visualiser hero section. Instead of a simple opacity crossfade, the transition between before/after images uses a construction teardown/assembly effect — fragments of the image break apart and reassemble like building materials.

**Slider-driven.** The existing drag handle controls the animation. As the user drags from left to right, the "before" image fragments scatter and the "after" image assembles. Subtle, elegant, professional.

**Platform-wide.** Available to ALL builds: tenant-builder batch, warm-leads, norbot-showcase. Not tenant-specific.

## How It Works

### Image Configuration

The component reads `heroVisualizerImages` from the tenant's `company_profile`:

```typescript
interface HeroVisualizerImages {
  before: string;
  styles: Array<{ label: string; after: string }>;
}
```

Falls back to hardcoded defaults if not set:
- Before: `/images/hero/before-kitchen.png`
- Styles: Modern, Farmhouse, Industrial

Images are generated by `tenant-builder/lib/hero-image-generator.mjs` using Gemini 3.1 Flash.

### Fragment System

`fragment-utils.ts` provides:
- `generateFragments(count, seed)` — Creates N irregular polygon fragments via Voronoi tessellation
- `toClipPath(vertices)` — Converts vertices to CSS `polygon()` string
- `interpolateFragment(position, fragment, phase)` — Computes transform values for a given slider position

Fragments are deterministic (seeded PRNG). Same fragments every render. No visual jitter.

### Animation Flow

1. **Mount:** Auto-intro animation reveals the slider (same entrance as existing visualizer-showcase)
2. **Idle:** Before image fully visible (slider at 0%)
3. **Drag 0→100%:** Before fragments scatter outward → After fragments assemble inward
4. **Style tab switch:** Changes the "after" image, re-renders fragments with new source
5. **Reduced motion:** Falls back to standard opacity slider (no fragments)

### Interaction Model

- Mouse drag on slider track (same as existing)
- Touch drag on mobile (same as existing)
- Keyboard: Arrow keys (10% steps), Home (0%), End (100%)
- Auto-intro: Animates 0→100→88% on mount (above the fold)

## Adding to a Tenant

The tenant-builder architect can select this hero variant in the SiteBlueprint:

```
homepage_layout: ['hero:visualizer-teardown', 'nav:sticky-simple', ...]
```

For existing tenants, update the `homepage_layout` in `admin_settings` to swap `hero:visualizer-showcase` for `hero:visualizer-teardown`.

## Modifying the Animation

### Changing Fragment Count
In the component, adjust the `generateFragments()` call:
```typescript
const fragments = useMemo(() => generateFragments(20, 42), []); // count, seed
```
- Fewer fragments (8-12): Larger pieces, more "construction material" feel, better mobile performance
- More fragments (16-24): Finer shatter, more dramatic, may need mobile testing

### Changing Scatter Intensity
In `fragment-utils.ts`, adjust the `scatterMagnitude` in `generateFragments()`:
```typescript
const scatterMagnitude = 0.5 + distFromCentre * 1.5; // Increase 1.5 for more dramatic scatter
```

### Changing Animation Easing
In `interpolateFragment()`, the `smoothstep` function controls the easing curve. Replace with other curves as needed:
```typescript
const eased = t * t * (3 - 2 * t); // smoothstep — smooth acceleration/deceleration
```

## Performance Requirements

- **GPU-safe:** Animate ONLY `transform` and `opacity` (per taste-skill §5)
- **60fps on mobile:** Test at 375px. Fragment count should stay <=20 for safe mobile performance.
- **will-change:** Apply `will-change: transform` selectively (only during active drag)
- **Framer Motion only:** No GSAP. taste-skill explicitly says "NEVER mix GSAP with Framer Motion in same component tree."
- **Isolated client component:** The animated fragment container should be its own `'use client'` leaf component to avoid re-rendering the parent layout.

## Accessibility

- `useReducedMotion()` → standard opacity slider (no fragments, no shatter)
- Keyboard support: Arrow keys, Home, End (same as existing)
- Screen reader: Before/After labels, ARIA attributes on slider
- Aspect ratio: 16/10, responsive sizing (same as existing)
