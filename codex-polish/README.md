# Codex Tenant Polish Workspace

This folder is the **Codex-only polish workspace** for ConversionOS demo tenants. Use it with the OpenAI Codex app (or CLI) to refine a built tenant so it matches the original company's brand (logos, colours, copy, photos).

## Prerequisites

- A tenant already built and deployed (e.g. `https://acme.norbotsystems.com`).
- The **original company URL** (e.g. `https://acme-renovations.com`) where copy and branding were sourced.
- Codex opened at the **demo repo root** and run from this folder (e.g. `codex --cd codex-polish`).

## Running a polish session

1. **Open the demo repo in Codex** (app or CLI from repo root).

2. **Start from this folder**  
   - CLI: `codex --cd codex-polish`  
   - In the app: set task context to `codex-polish` or paste instructions that reference this folder.

3. **Read from the queue first (automation mode)**  
   If the tenant-builder has queued work for Codex, look in:
   - `codex-polish/queue/pending/<site_id>.json`

   Queue items include:
   - the live demo URL
   - the original company URL
   - QA verdict (`READY`, `REVIEW`, `NOT READY`)
   - result artifact paths (`go-live-readiness.json`, `audit-report.md`, `visual-qa.json`, `scraped.json`)

4. **Optional: dump current tenant state**  
   From repo root:
   ```bash
   node scripts/polish/dump-tenant.mjs --site-id <site_id>
   ```
   This writes `codex-polish/current-tenant.json` so Codex can read current branding/copy without Supabase access.

5. **Tell Codex** (in chat or in `input-template.md`):
   - **Demo URL:** `https://<site_id>.norbotsystems.com`
   - **Original URL:** `https://original-company.com`
   - **Site ID (optional):** if the subdomain isn’t obvious, pass it explicitly.

6. Codex will infer `site_id`, compare original vs current state, and either:
   - write **`codex-polish/patches/<site_id>.json`**, or
   - decide no patch is needed and run:
   ```bash
   node scripts/polish/complete-polish.mjs --site-id <site_id> --note "No changes needed"
   ```

7. **Apply the patch** (from repo root, with `.env.local` or pipeline env loaded):
   ```bash
   node scripts/polish/apply-polish-patch.mjs --site-id <site_id>
   ```
   This also clears the active polish queue item so outreach can proceed.

8. **Final check in Cursor**  
   Start the dev server, open `http://localhost:3000?__site_id=<site_id>`, and do the last 1–2% (visual tweaks, copy) before shipping.

## Files in this folder

| File | Purpose |
|------|--------|
| `AGENTS.override.md` | Instructions for Codex (role, workflow, boundaries). Do not edit unless you are changing the polish contract. |
| `AUTOMATION_PROMPT.md` | Prompt to paste into a Codex automation so it processes `queue/pending/` on a schedule. |
| `README.md` | This file. |
| `input-template.md` | Optional: paste Demo URL and Original URL here for Codex to read. |
| `queue/pending/<site_id>.json` | Active polish jobs produced by the tenant-builder after QA. Presence of this file blocks outreach. |
| `queue/done/<site_id>.json` | Archived queue items after polish is completed or skipped. |
| `current-tenant.json` | Generated by `dump-tenant.mjs`; optional input for Codex. |
| `patches/<site_id>.json` | Generated by Codex; consumed by `apply-polish-patch.mjs`. |

## Safety

- Codex is instructed **not** to edit `tenant-builder/`, `src/proxy.ts` (except when adding a tenant), or shared app code.
- All tenant-specific changes are **data only**: patch files applied via `apply-polish-patch.mjs` to Supabase `admin_settings` for that `site_id`.
- Active queue items in `queue/pending/` block outreach until they are cleared by `apply-polish-patch.mjs` or `complete-polish.mjs`.
- For extra safety, run Codex in a **worktree** so the main branch stays untouched until you and Cursor are satisfied.
