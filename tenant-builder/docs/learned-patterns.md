# Learned Patterns

Accumulated learnings from tenant builds. Read at session start. Append after corrections.

## How to Use This File

- **Read** this file at the start of every build session
- After fixing an issue in a tenant, ask: "Is this a pattern that will repeat for other tenants?"
- If yes, append it here with: what the issue was, what caused it, what the fix was, which build it came from
- Every ~10 builds, review for patterns worth codifying into the pipeline (propose a plan, don't auto-modify)
- Format: `[date] [site-id]: description of pattern`

## Hero Images

**[2026-03-03] mccarty-squared-inc:** Scraper captured the logo CDN URL as `hero_image_url` instead of the actual hero background photo. Result: `hero.png` in storage was the logo, and `company_profile.heroImageUrl` was an incomplete base URL (`...mccarty-squared-inc/` without filename). Fix: update `heroImageUrl` in Supabase to `portfolio/0.jpg` (the best renovation photo). Pattern: when `hero_image_url` from the scrape looks like a logo CDN path (contains "logo", "Logo", or "MSI"), discard it and use the first portfolio image as the hero instead.

**[2026-03-03] mccarty-squared-inc:** `company_profile` `heroImageUrl`, `logoUrl`, and `aboutImageUrl` fields were saved as the storage bucket base URL without a filename suffix (provisioning bug). All three were `...mccarty-squared-inc/` instead of `...mccarty-squared-inc/logo.png`. Likely a provisioner string concatenation bug. Fix manually via Supabase PATCH. Recommend: add URL validation in provision-tenant.mjs — if a URL ends with `/`, it's invalid.

## Colour & Contrast

**[2026-03-03] bl-renovations:** Logo was invisible in the header — white logo on white background. The scraper captured their white-on-dark version (`logo.webp` — 600x600, white text on light grey). Their actual website uses a dark/black version for light backgrounds. Fix: downloaded dark logo from their site header, uploaded as `logo-dark.png` to Supabase Storage, updated `company_profile.logoUrl` and `branding.logoUrl`. Pattern: when scraping logos, check if the logo has transparency (PNG/WebP with alpha). If so, check if the text/graphic is light-coloured — it may be a "logo on dark" variant. Always try to source both light and dark versions. If only a white logo is available, set `branding.logoOnDark: true` so the header wraps it in a dark pill.

## Logo Sizing & Visibility

**[2026-03-03] all tenants:** Platform header constrained logos to 32px (`h-8`) in a 64px header — wasting half the available space. All tenants looked undersized. Fixed platform-wide: header now `h-10 md:h-11` (40px mobile, 44px desktop), footer `h-10` (40px). Image attrs increased to 220×56 / 200×50 for retina sharpness. Text fallback scaled from `text-xl` to `text-2xl`. Dark pill padding increased from `py-1.5` to `py-2`. Pattern: this is a platform-level concern, not a per-tenant issue. Future builds don't need to adjust logo sizing. However, scraped logos should be minimum 200px wide for sharp rendering at 44px display height. Visual QA should verify logo is VISIBLE (not just loaded) — white logos on white backgrounds pass "image loaded" checks but are invisible.

## Copy & Content

**[2026-03-03] mccarty-squared-inc:** Scraper hit their Strathroy-specific subpage (`mccartysquared.ca/strathroy-ontario`) instead of the root. All scraped about copy referenced "Strathroy community" when the company's primary address is London, ON. Fix: manually patched `aboutCopy` in Supabase to say "London and surrounding areas". Pattern: always scrape from the root domain, not regional subpages. Consider checking `source_url` in `_meta` — if it contains a city slug, re-scrape from root.

## Services & Portfolio

**[2026-03-04] brouwer-home-renovations:** `filterServices` silently dropped all 7 services because they had empty descriptions (gate requires `description.length >= 10`). Result: `company_profile.services = []` and the homepage services section was completely blank. The descriptions DO exist — they're on each `/service-name` sub-page as the hero paragraph — but Firecrawl only scrapes the homepage and doesn't follow service sub-page links. Fix: after any build where services count is 0, manually scrape each service sub-page for the description paragraph and re-provision. Long-term: scraper should visit each detected service link and capture its description.

**[2026-03-04] brouwer-home-renovations:** Service page background images are the highest-quality photos for each service — far better than generic gallery images. Each background is confirmed by page context (page title = service name = image subject). Pattern: when a contractor has per-service sub-pages, scrape those backgrounds as `imageUrl` for each service in `company_profile.services`.

**[2026-03-04] general — portfolio description integrity:** Gallery images on contractor sites typically have NO individual labels (alt text is the business name). Any specific descriptions like "— Vanity & Mirror" or "— Glass Shower Enclosure" are AI-guesses and must NOT be used. Correct approach: titles should be generic room-type only ("Bathroom Renovation") and `description` should be empty string unless the original site explicitly labels that photo. Never add detail suffixes without a confirmed source URL.

**[2026-03-03] mccarty-squared-inc:** Page completeness checker false-positive FAILs on `/services` (0 cards) and `/projects` (no items) even though all content rendered correctly. Root cause: selectors used class-name substring matching (`[class*="service"] [class*="card"]`) which doesn't match shadcn/ui components — these use `data-slot="card"` with Tailwind utility classes, no "card" in class names. Fix: use `[data-slot="card"]` selector in page-completeness.mjs. Also: `img.naturalWidth > 0` returns 0 for lazy-loaded images; use `img.srcset || img.src` instead.

## Certifications & Memberships

**[2026-03-03] bl-renovations:** About page showed hardcoded RenoMark benefit cards (2-Year Warranty, $2M Insurance, Code of Conduct, Written Contracts, 2-Day Response) even though BL Renovations only has "Commercially Bonded" and "Insured" — they are NOT a RenoMark member. This is false certification and could be legally problematic. Root cause: the About page rendered RenoMark details unconditionally for any tenant with certifications. Fix: gated the RenoMark details section behind `config.certifications.some(c => /renomark/i.test(c))`. Pattern: certification details must NEVER be shown unless the tenant's scraped data explicitly includes that certification. Only McCarty Squared Inc. has RenoMark. The scraper must capture exact certification names and the platform must gate org-specific benefit claims behind presence of that certification.

## Scraping Edge Cases

**[2026-03-04] brouwer-home-renovations:** Gallery was hidden under Resources → Gallery (`/gallery`), not linked from the homepage. Firecrawl only follows homepage links so it missed 18 project photos entirely. Scraped portfolio = 0. When `portfolio.length === 0` after a build, check if the site has a `/gallery`, `/portfolio`, or `/projects` page under a secondary nav menu and scrape it separately. This is a common pattern for sites built on OneLocal/Duda/Wix.

**[2026-03-04] brouwer-home-renovations:** Hero image was the logo — both `hero_image_url` and `logo_url` resolved to the same CDN path (`Brouwer-Home-Renovations-LogoBlack-68eca2be-1920w.jpg`). The actual site hero is a CSS `background-image` property, not an `<img>` tag. Firecrawl's v2 branding extractor saw the logo `<img>` tag and set both fields to it. Fix: after provisioning, verify that `heroImageUrl !== logoUrl`. If they match, it's a scraping confusion — look for CSS background images in the hero section instead.

**[2026-03-03] general:** If `_meta.source_url` in the scraped JSON contains a city slug (e.g. `/strathroy-ontario`, `/london-ontario`), the scraper hit a regional landing page rather than the root. City-specific pages have narrower content (only mentions that city). Always prefer the root URL for the primary scrape.

**[2026-03-07] sonce-homes:** Gallery was at `/our-portfolio/` (not `/gallery`, `/portfolio`, or `/projects`). Linked from footer but not from homepage nav. Firecrawl scraped 0 portfolio images but the actual page had 68 real renovation photos. Pattern: when `portfolio.length === 0`, scan ALL footer hrefs — the gallery may use a custom slug like `/our-portfolio/`, `/our-work/`, `/recent-projects/`. Check any footer link whose text or path contains "portfolio", "gallery", "projects", "work", or "photos".

**[2026-03-07] sonce-homes:** Scraper wrote a "Houzz" social with `href = "https://soncehomes.ca/"` — the company's own website URL, not a Houzz profile. This happens when the scraper sees a Houzz badge image but no Houzz link, and falls back to the page URL. Fix: filter socials where the href domain matches the company's own domain.

**[2026-03-07] sonce-homes:** Google Reviews via Trustindex widget are fully scrapeable via curl + HTML. Reviewer names are in `class="ti-name"` elements; review body text follows "Trustindex verifies that the original source of the review is Google." as a splitter. Check `/about-us/` page for Trustindex reviews on WordPress contractor sites.

## Email & Outreach

**[2026-03-04] general — ferdie@norbotsystems.com showing on contact pages:** When `business_info.email` is empty string, `branding.ts` and `branding-provider.tsx` fell back to `DEMO_BRANDING.email` (`ferdie@norbotsystems.com`) because `'' || DEMO_BRANDING.email` evaluates to the fallback (empty string is falsy). Fixed by changing `|| DEMO_BRANDING.email` to `|| ''` for email/paymentEmail/quotesEmail fields. Pattern: after building any tenant, verify the Contact page shows the right email — not ferdie's. If empty, that's fine; if ferdie's, the DB email field is empty and the code fallback fired.

**[2026-03-04] general — base64 hero in company_profile.heroImageUrl:** If the provisioner scrapes a hero that is embedded as a CSS `background-image` with a data URI, it may write the base64 blob directly to `heroImageUrl`. This makes the field unrenderable as an `<img>` src in the platform. Detection: if `heroImageUrl` starts with `data:` or the DB returns `<Base64-Image-Removed>` in output, it needs replacement. Fix: find a real image on the site, download it, upload to Supabase storage, update heroImageUrl to the storage URL.

**[2026-03-04] general — "Not available" in socials array:** Some scrapers write `{href: "Not available", label: "Facebook"}` when a social link doesn't exist, instead of omitting the entry. These render as broken links in the footer. Fix: filter `socials` to exclude entries where `href === "Not available"` or contains "not available".

**[2026-03-04] general — AI-hallucinated service packages and imageUrls:** Service `packages` arrays (with `name`, `description`, `startingPrice`) are often entirely hallucinated by the LLM during provisioning — the contractor's website has no such content. Similarly, `imageUrl` fields like `joescarpentry.ca/services/renovation1.jpg` are fabricated paths that return 404. Identify by checking: does the URL resolve? Is the pricing plausible for the service? Hallucinated packages should be cleared to `[]`. Broken imageUrls should be replaced with real uploads or set to `''`.


**[2026-03-05] general — principals field stored as array:** When `data.principals` from scraping is an array (e.g. `['Adam']`), `provision.mjs` assigns it verbatim to `company_profile.principals`. The About page `getInitials(name.trim())` then crashes with "name.trim is not a function" → HTTP 500 on `/about`. Fix: check if `company_profile.principals` is an array in Supabase; if so, PATCH it to a comma-joined string. Prevention: `provision.mjs` should join principals arrays before storing (e.g. `Array.isArray(p) ? p.join(', ') : p`).

**[2026-03-05] general — Playwright fullPage screenshot blank sections (false positive):** Full-page screenshots taken with `fullPage: true` after `waitUntil: 'networkidle'` sometimes show large blank white voids between the hero/stats and the footer. All mid-page sections (AI visualizer teaser, services, testimonials) appear missing. Root cause: `fullPage: true` extends the viewport height but does NOT scroll the page, so IntersectionObserver-based animations/reveals stay in their initial (hidden) state for off-screen content. Fix applied to `qa/screenshot.mjs`: scroll through the page using `requestAnimationFrame` to bottom then back to top before taking the full-page screenshot. This triggers all IntersectionObserver callbacks. VQA scores for layout_integrity and brand_cohesion were penalised by this false positive — if a site scores low on layout with a "blank section" complaint but renders correctly in browser, this is the cause.

**[2026-03-05] general — VQA subprocess unavailable in nested Claude session:** `visual-qa.mjs` invokes `claude -p ... --model sonnet` as a subprocess. When called from within an active Claude Code session (CLAUDECODE env var set), deleting CLAUDECODE and calling the nested claude binary results in "Command failed" with no visible error. The VQA result file is not updated. Workaround: assess VQA scores manually from screenshots when running in a nested session. Long-term fix: replace `execFileSync claude -p` with direct Claude API call via the Anthropic SDK.

**[2026-03-05] general — Google social link cross-contamination:** Scraped `branding.socials` may contain a Google Maps/Business link that resolves to a different company entirely (e.g. lenoircitypainting.com appearing on a-and-a's Google social). Firecrawl may capture nearby Google Maps embed links or neighbourhood business links. Always validate social link domains against the company's own domain and city. If the href domain doesn't match the company, remove the entry.

**[2026-03-07] ancaster-home-renovations — Web-share footer links scraped as social media:** Old-style sites (pre-2020 static HTML) often have a "Web Share" footer section listing partner/affiliate websites for SEO link-exchange purposes. Firecrawl may capture these as legitimate social links if they are grouped near the footer social icons. Example: tampabayepoxyflooring.com was stored as the "Google" social because it appeared in the footer alongside Facebook/Twitter links. Fix: filter `branding.socials` to only keep entries where the href domain is a known social platform (facebook.com, twitter.com, instagram.com, linkedin.com, houzz.com, etc.) OR matches the company's own domain. Remove any commercial website URLs masquerading as socials.

**[2026-03-07] lrc-construction — White-on-dark logo with no alternate version:** Some contractors (especially those using WordPress/Elementor with a dark header) only have one logo file — white graphic on dark background JPEG. Unlike PNG/WebP with transparency, this is an opaque image that renders entirely wrong on a white header background. Detection: logo JPEG is dark-tinted with white text/icon. Fix: upload the logo as-is to Storage and set `branding.logoOnDark = true`, which wraps it in a dark pill in the header. Do NOT try to invert or reprocess the image. This differs from the BL Renovations transparent-logo pattern — there it was a white logo with alpha; here it's a dark background baked into the JPEG.

**[2026-03-07] devrye-custom-renovations — Gallery images on Squarespace sites:** Squarespace sites use `images.squarespace-cdn.com` CDN with UUID-based paths. The scraper may miss gallery pages entirely because they are behind `/galleries`, `/kitchens-gallery`, `/bathrooms-gallery`, `/basements-gallery` etc. (per-category gallery pages linked from nav only). Fix: check the nav links for `-gallery` suffix URLs, scrape each one separately. Squarespace image URLs follow the pattern `https://images.squarespace-cdn.com/content/v1/{site-id}/{uuid}/{filename}` — these are stable and directly downloadable. Always prefer the non-thumbnail version (no `?format=Xw` suffix) for the highest quality.

**[2026-03-07] graham-s-son-interiors — Original site down at time of fix (503):** When the target's original website is unavailable (503/maintenance), services and testimonials can still be recovered from the previously stored `scraped-raw.json` in `tenant-builder/results/{date}/{site-id}/`. The file contains all raw scraped fields including `services[]`, `testimonials[]`, `about_copy[]`, and `portfolio[]`. Service descriptions can be derived from `about_copy` and `why_choose_us` fields if sub-page content was not captured. Pattern: always try `results/{date}/{site-id}/scraped-raw.json` before declaring a scrape impossible. Portfolio image gallery-type grouping (from original CDN paths in scraped-raw) can be used to map already-uploaded storage images to appropriate service imageUrls.

**[2026-03-07] ancaster-home-renovations — Testimonial author/quote mismatch from scraping:** The provisioner stored a testimonial with `author: "Jonathan Manzine"` (the contractor's name) paired with a testimonial quote that was actually from "Justin Bester". When scraping testimonials, the author and quote fields can get misaligned, especially on sites using carousel/slider markup where the author is in a separate `<h2>` element below the `<p>` quote. Always verify testimonial author names are real clients (not the business owner) and that quote text matches the attributed author. Cross-reference against the original testimonials page.

**[2026-03-07] ancaster-home-renovations — Slider images as hero source for image-poor static sites:** Old static HTML sites using Revolution Slider or bxSlider often have their best renovation photos as slider background images (`<img src="images/slide01.jpg">`) rather than in a gallery page. The scraper may miss these entirely if it only extracts `<img>` tags from the content area. Pattern: when `portfolio.length === 0` and the site has a slider/banner section, check for `images/slide0N.jpg` or similar slider image paths — these are the best hero candidates.

**[2026-03-07] hemeryck-homes-construction-ltd:** Scraper hit the HNHBA directory listing (hnhba.com/members/...) instead of the contractor's own website (hemeryckhomes.ca). The directory page showed their brand colour as #ff6900 (orange from the WordPress theme palette) instead of their actual green (#6bba57, palette2 from their own site). Fix: always check if `scraped.json._meta.source_url` is a third-party directory page — if so, scrape from the contractor's own website for colours, images, and services. The HNHBA directory was the provided "original site" URL which caused the confusion.

**[2026-03-07] hemeryck-homes-construction-ltd:** Google Reviews shown via Elfsight JS widget (`elfsight-app-*`) cannot be scraped — the widget loads reviews client-side via authenticated API calls. When a site uses Elfsight for reviews, testimonials will always be empty after scraping. Workaround: set `trustMetrics.google_rating` and `trustMetrics.projects_completed` from the AggregateRating schema.org data visible in the page HTML (which IS accessible). Flag to Ferdie that testimonial quotes must be sourced manually from their Google Business profile.

**[2026-03-07] general — hero from YouTube video background:** Some WordPress/Elementor sites use a YouTube video as the hero background rather than a static image. The scraper cannot capture this and may fall back to the brand logo image (square format). Detection: if `hero.jpg` in storage is square (same width and height) and under 200KB, it's likely the brand logo image rather than a project photo. Fix: download a landscape project photo from their gallery and re-upload as hero.jpg.

**[2026-03-07] tyton-homes — TrustIndex Google Reviews widget has full review text in page HTML:** WordPress sites using the TrustIndex plugin (trustindex.io) embed a `<template>` block with the full review text and author names as a pre-rendered HTML blob inside a `<pre>` tag. Unlike Elfsight (which loads reviews client-side via API), TrustIndex pre-renders review content server-side into the HTML. Pattern: when `grep -i 'ti-review-item\|trustindex' page.html` shows matches, parse the `<!-- R-CONTENT -->...<!-- R-CONTENT -->` sections for quote text and `ti-name` divs for reviewer names. All 6 reviews from Tyton Homes were extracted this way from tytonhomes.com.

**[2026-03-07] way-mar-home-renovation-contractors — Squarespace gallery page has best portfolio images:** Squarespace sites store images at `https://images.squarespace-cdn.com/content/v1/{site-uuid}/{upload-id}/{filename}`. The gallery page (`/gallery`) uses these CDN URLs directly in `<img>` tags. No query params needed for full resolution — just the base URL returns the full image. When a Squarespace contractor site has zero portfolio images after provisioning, check `/gallery` separately with a direct curl — Firecrawl may have only crawled the homepage. Pattern: look for images matching the `images.squarespace-cdn.com/content/v1/` pattern with filenames that are 6-character alphanumeric codes (e.g. `D71_7789.JPG`, `JK2_3749.JPG`) — these are professional photography uploads, not stock images.

**[2026-03-07] ancaster-home-renovations — Old static HTML site hides gallery under `images/gallery/` subfolder:** The site's `photos.html` page referenced images as `src="images/gallery/1s.jpg"` but the scraper only found `images/slide01.jpg` etc. from the homepage. The gallery images (`1s.jpg` through `19s.jpg`) were in a subdirectory not obvious from the homepage. Fix: always check if a static HTML site has a `photos.html` or `gallery.html` page, and read the actual `src` attributes to discover the correct image directory. Do NOT guess `images/` — it may be `images/gallery/`, `images/projects/`, `images/photos/`, etc.

**[2026-03-07] general — Brand logos (electrical, material brands) can appear as project photos:** Some renovation contractors (especially electricians) have a "Brands We Work With" or "Partners" section with vendor logos (Eaton, Siemens, Westinghouse, GE, Square D, etc.). These are small PNGs in the `images/` folder with a naming pattern like `emer_r.png`, `ge_r1.png`, `square_d_r1.png`. The scraper picks them up alongside renovation photos. Detection: if filenames end with `_r.png` or `_r1.png`, or match known brand names (emer, westinghouse, siemens, cutler, sylvania, allen_bradley, s&c_electric), they are brand logos — NOT renovation photos. Fix: filter these from the portfolio before uploading.

**[2026-03-07] hemeryck-homes-construction-ltd — WordPress/Elementor sites serve images in AVIF format:** Modern WordPress with Smush/ShortPixel image optimization plugins auto-generate AVIF versions (`*.png.avif`, `*.jpg.avif`). When Elementor loads images, it serves the AVIF version via URL rewriting. Node's `fetch` without special Accept headers may get AVIF responses that fail our upload (MIME type `image/avif` rejected by Supabase). Fix: use `Accept: image/jpeg,image/png,image/webp,*/*;q=0.8` (no `image/avif`) in the fetch headers, and always use the original JPG/PNG URL from `data-src` rather than the AVIF URL from `src`. As a fallback, use the HNHBA directory page images (which are served as JPEG) when the contractor's own site serves only AVIF.

## Data Quality Hardening (6 issues codified — Mar 8, 2026)

These 6 patterns were discovered during manual polish of 13 demo tenants on Mar 7, 2026 and codified into the pipeline as pre-validation quality gates on Mar 8, 2026.

**[2026-03-08] Issue #1 — Testimonial field name mismatch:** Scrapers produce `name`/`text`/`type` but `filterTestimonials()` expected `author`/`quote`/`project_type`. Testimonials silently dropped to `[]` for every build. Fix: added `normalizeTestimonial(t)` to `quality-gates.mjs` that maps variant field names to canonical form. `filterTestimonials()` now normalizes before filtering. Affected files: `quality-gates.mjs`.

**[2026-03-08] Issue #2 — aboutCopy type coercion:** Scraper sometimes returns `about_copy` as a string instead of `string[]`. Platform's `.map()` call crashes on a bare string. Fix: added `normalizeAboutCopy(raw)` to `quality-gates.mjs` — coerces string/null/undefined to `string[]`. `provision.mjs` now uses `normalizeAboutCopy(data.about_copy)` instead of `data.about_copy || []`. Affected files: `quality-gates.mjs`, `provision.mjs`.

**[2026-03-08] Issue #3 — Hotlinked hero/logo/about images not cleared on upload failure:** `upload-images.mjs` correctly cleared portfolio images to `''` on download failure (line 255) but kept broken hotlink URLs for hero, logo, and about images. Fix: added `else` branches that clear the URL and log the failure, matching the portfolio pattern. Added `failedUploads[]` tracking array and `_upload_failures` metadata in output JSON. Affected files: `upload-images.mjs`.

**[2026-03-08] Issue #4 — Generic portfolio title repetition:** `filterPortfolio()` only checked `title.length > 0`. Scraper generates "Custom Renovation" for every item. Fix: added `diversifyPortfolioTitles(portfolio)` to `quality-gates.mjs` — if >50% share the same title or are generic (10 known patterns), replaces with `service_type`/`room_type`-specific titles, fallback to `"Project N"`. `filterPortfolio()` now calls `diversifyPortfolioTitles()` after filtering. Affected files: `quality-gates.mjs`.

**[2026-03-08] Issue #5 — Foreign brand name leakage detection:** Content integrity only checked hardcoded demo strings. Doesn't detect other contractors' names in a tenant's scraped data (e.g., "Manzine Contracting" appearing in Ancaster's about copy). Fix: added `detectForeignBrandNames(data, expectedName)` to `quality-gates.mjs` — regex matches contractor-name patterns that don't match the expected business name. Warning-only (not blocking) to avoid false positives. Also added `checkForeignBrandNamesOnPage()` to `content-integrity.mjs` (Check 13) for post-provision rendered-page verification. `orchestrate.mjs` now passes `--business-name` flag to content-integrity. Both functions include false-positive filtering: pronoun/article prefix skip, preposition-ending prefix skip, word-overlap check vs expected name. Affected files: `quality-gates.mjs`, `provision.mjs`, `content-integrity.mjs`, `orchestrate.mjs`.

**[2026-03-08] Issue #6 — Upload failure logging:** Portfolio items with failed uploads were silently dropped. No record of what was lost. Fix: combined with Issue #3 — `failedUploads[]` array tracks all failures (hero, logo, about, portfolio[N]), prints summary, writes `_upload_failures` metadata to output JSON for downstream consumption. Affected files: `upload-images.mjs`.

## Bespoke Pipeline (March 2026)

**[2026-03-10] general — Codex --image flag causes 10min+ timeouts:** When `codex exec --full-auto --image` is used with screenshots, GPT 5.4 xhigh takes 10+ minutes and frequently times out at 600s. Without images, sections generate in 30-120s. Root cause: vision processing + xhigh reasoning + large prompt = API timeout. Fix: skip images entirely — CSS tokens + HTML reference provide sufficient visual context. Text-only Codex is 5-10x faster with comparable quality. Affected files: `build-custom-sections.mjs`.

**[2026-03-10] general — ContentElevation schema too strict for Opus output:** Opus 4.6 produces `contentElevation` entries where `current` is sometimes a number or boolean (e.g., `current: 3` for star rating, `current: false` for a flag). Also omits `field` key occasionally. Zod schema rejected with "Invalid input: expected string". Fix: use `z.coerce.string()` for `current` and `suggested`, and `z.string().optional().default('unknown')` for `field`. This prevents fallback to the generic 5-section blueprint. Affected files: `schemas/site-blueprint-v2.zod.mjs`.

**[2026-03-10] general — Foreign brand name detector false positives (71-74 per build):** The content-integrity `checkForeignBrandNamesOnPage()` fires on legitimate contractor content — service names, SEO meta, taglines, CTA text. "Bathroom Renovation", "Request a Free Estimate", "Port Stanley ON Renovation Contractor" all trigger it. These are NOT foreign brand leakage. Fix needed: tune the detector to only flag when another contractor's actual business name appears, not generic renovation industry language.

**[2026-03-11] FIX APPLIED — false positives reduced via whitelist + warning-only:** Added `RENOVATION_INDUSTRY_TERMS` Set (50+ entries) and `ADJECTIVE_PREFIXES` regex (30+ patterns) to `content-integrity.mjs`. Foreign brand name matches against these are skipped. Changed Check 13 to `foreign_brand_name_warning` (excluded from scored violations, doesn't affect QA verdict).

**[2026-03-10] general — Codex explores project CLAUDE.md before writing:** Without directive prompts, Codex spends 5-10 minutes reading ConversionOS project files (CLAUDE.md, package.json, etc.) before writing the section file. Fix: prepend `"IMPORTANT: Create the file IMMEDIATELY. Do NOT read other project files."` to the prompt. This reduces generation time from 10min+ to 30-120s. Affected files: `build-custom-sections.mjs`.

**[2026-03-10] general — CSS token extraction fails when dynamically imported:** `css-extract.mjs` had `parseArgs()` at module top-level, which crashes when the file is dynamically imported by `scrape-enhanced.mjs` (sees parent's CLI flags as unknown args). ## Session 25 Learnings (2026-03-15) — First Warm-Lead Pipeline Run

**[2026-03-15] Firecrawl v4 API breaking change:** `@mendable/firecrawl-js` v4.13.0 moved `scrapeUrl()` from the root instance to `.v1.scrapeUrl()`. The pipeline hangs silently when calling the non-existent method. Fix applied to `scrape-enhanced.mjs`: `const firecrawl = fcInstance.v1 ?? fcInstance`. Already fixed in `lib/firecrawl-client.mjs` (line 77). **Permanent fix: after ANY npm update, verify Firecrawl API surface.**

**[2026-03-15] @anthropic-ai/sdk not in package.json:** The tenant-builder imports `@anthropic-ai/sdk` (via `lib/anthropic-client.mjs` and `qa/content-fidelity.mjs`) but it was never added to `package.json`. Fixed by `npm install @anthropic-ai/sdk`. **Permanent fix: add to package.json devDependencies.**

**[2026-03-15] Playwright browsers not installed:** All browser-based QA modules fail silently with "Executable doesn't exist" error. Fixed by `npx playwright install chromium`. **Permanent fix: add to setup checklist and pre-flight check in orchestrate.mjs.**

**[2026-03-15] False READY verdict when all QA checks are null:** When Playwright is not installed, all QA modules return null. The audit-report.mjs defaults to READY with `human_review_required: true`. This is dangerous — a build with zero QA should never be READY. **Permanent fix: audit-report should check for null/undefined checks and verdict NOT READY if >3 checks are null.**

**[2026-03-15] aboutCopy type crash in original-vs-demo:** `about_copy` field can be string or string[] depending on the scraper. `scrapedAboutCopy.toLowerCase()` crashes on arrays. Fix applied: normalise array to joined string at function entry. **Permanent fix: normalise all text fields at scrape time, not at QA time.**

**[2026-03-15] Scrape reuse skips warm-lead fresh data:** When `scraped.json` exists from a previous build, the pipeline skips scraping entirely. For warm-leads, this means stale data is used. **Permanent fix: add `--force-scrape` flag or make warm-lead mode always re-scrape.**

**[2026-03-15] Opus 4.6 text-only architect timeout (300s):** Both attempts timed out for CCR Renovations (a straightforward site). The static fallback blueprint was used instead. This is consistent with previous sessions. **Recommendation: increase timeout to 600s for warm-lead mode, or skip Opus and use Claude Code SDK for in-process architect instead of spawning a claude -p subprocess.**

**[2026-03-15] Codex CLI not available:** Content architect fails because `codex exec` is not installed or rate-limited. Falls back gracefully to manual content extraction from scraped HTML. **Recommendation: verify codex CLI subscription status; if unavailable, use claude -p as the primary content extractor.**

**[2026-03-15] Hero image generation partial:** Gemini 3.1 Flash generated 2 of 3 requested styles (Modern + Industrial, Farmhouse silently failed). Images are 1.7-1.9MB each, uploaded to Supabase Storage successfully. **Recommendation: add retry for failed styles; 2/3 is acceptable but 3/3 is ideal.**

**[2026-03-15] Live site audit 7/8 pass:** Only SEO meta failed (missing description or og:image). This is a provisioning gap — meta tags need to be seeded in admin_settings during provision step. **Permanent fix: add og:title, og:description, og:image to provision pipeline.**
